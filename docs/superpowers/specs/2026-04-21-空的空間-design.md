# 空的空間（The Empty Space）

**雙角色觀察實驗台 — Design Spec**

- 作者：陳柏為 × Claude（蒼鷺）
- 日期：2026-04-21
- 狀態：design draft，待 review

---

## 0. 命題

> "I can take any empty space and call it a bare stage. A man walks across this empty space whilst someone else is watching him, and this is all that is needed for an act of theatre to be engaged."
> — Peter Brook, *The Empty Space* (1968)

**兩個演員 + 一個看著的人 = 最小戲劇單位。**

這個 project 把這句話變成 engine：兩個由方法論塑形的 LLM 角色在空的空間裡對話，觀察者（柏為 + Claude）看著他們，engine 把過程全部落盤，dashboard 讓兩個觀察者看同一份真相。

它不是產品，是**實驗台**。目的是驗證「貫通軸 × 關係層 × Setting」穿過不同模型權重時能不能穩定地塑形成同一個角色。

---

## 1. 範圍

### In（MVP 做）

1. 雙角色自動對話 runner（Python orchestrator）
2. Layer 1 Prompt 組裝（貫通軸 + 關係層 + Setting + ambient echo）
3. Layer 2 Judge（stage / mode / 張力追蹤 / fire_release / basin_lock）
4. Layer 3 Composer（Opus 4.7 bake snapshot + hash cache）
5. Layer 4 印象帳本（按關係分組 YAML + symbol index + retrieval worker + 六維度 rubric）
6. 關係層 YAML → Claude Code skill 的 bootstrap 轉換腳本
7. 實驗配置檔 schema（protagonist_opener + initial_verb/stage/mode + scripted_turns + counterpart_system）
8. 所有中間產物落盤到 `runs/<exp_id>/`
9. 共同可觀察 dashboard（單 HTML + vanilla JS，讀檔呈現）

### Out（不做）

- 多角色廣場（N×N 配對載入器、場域 composer）
- 玩家 UI 模式切換（旁觀 / 插話 / 扮演 三檔）
- 真人玩家下場扮演某角色
- GLSL 視覺 / 成品 UI / 動效
- 從 transcript 萃新關係層 skill（hackathon pipeline 延伸，留給下一個 sub-project）
- Claude Code SessionStart hook 注入（那是 CLI 版的做法，這邊不用）
- 跨實驗的統計分析 / 評分 dashboard（現在先累積材料，分析之後寫別的工具）

## 3. 對話流程

### 3.1 驅動機制 (δ)

```
情境種子（γ）起手
  + 輪流推進（α）
  + scripted injection 硬觸發點
  + Judge fire_release / basin_lock 判定結束
```

### 3.2 單次實驗的生命週期

```
[實驗配置檔 experiment.yaml]
        ↓
   [Composer bake]  ← Opus 4.7
   產 snapshot.yaml（21 格 + effective 關係層 + Judge principles）
        ↓
   ┌────────── Turn loop ──────────┐
   │                                │
   │  [Layer 1 Prompt 組]           │
   │   貫通軸 + 關係層 + Setting    │
   │   + ambient echo from 帳本     │
   │        ↓                       │
   │  [角色 inference]  ← Gemini Flash
   │   產對話 + 候選印象句         │
   │        ↓                       │
   │  [Judge update]  ← Gemini Flash
   │   更新 stage / mode / 張力     │
   │        ↓                       │
   │  [scripted injection check]    │
   │   若到觸發點則硬注入           │
   │        ↓                       │
   │  [終止檢查]                    │
   │   fire_release? basin_lock?    │
   │   max_turns？                  │
   │                                │
   └─── no: 下一個角色 Turn ────────┘
                   ↓ yes
   [Session end]
        ↓
   [Rubric 評分]  ← Opus 4.7
   候選印象 → 六維度打分 → 進/不進帳本
        ↓
   [帳本寫回]
   A_x_B.impressions.yaml 更新（ledger_version bump）
        ↓
   [Composer pre-selection]  ← Gemini Flash
   從 ledger 挑 top-N 相關印象（symbol 命中 × 承重分數）
        ↓
   [Composer re-bake]  ← Opus 4.7
   讀預篩後的輸入 → 產 21 格 snapshot（下次 session 用的 cache）
```

### 3.3 情境種子（γ）的 schema

沿用你 `experiment_baseline.yaml` 的結構，extend 成正式版：

```yaml
exp_id: mother_x_son_hospital_v3_001
protagonist:
  path: 六個劇中人/母親
  version: v3_tension
counterpart:
  path: 六個劇中人/兒子
  version: v3_tension
setting:
  path: 六個劇中人/環境_醫院.yaml
protagonist_opener: |
  你是一個中年女人...（同 baseline 的 opener 邏輯）
counterpart_system: |
  你是一個年輕男人，面對你的親生母親...
  （沿用 baseline experiment_baseline.yaml 的 counterpart_system 寫法）
initial_state:
  verb: 承受（靠近）
  stage: 前置積累
  mode: 基線
scripted_turns:
  8:
    speaker: counterpart
    content: "你從來沒有找過我。不是他不讓你找。是你沒有找。"
max_turns: 20
termination:
  on_fire_release: true
  on_basin_lock: true
```

### 3.4 Scripted Injection 的角色

是**實驗嚴謹性的一部分**，不是作弊。v1 baseline 裡 Turn 8 的那句話是「共同事件」，讓所有跑出來的對話都有同一個張力峰值可比較。未來要做 A/B 對比（方法論 vs baseline、Gemini vs Opus），沒有硬觸發點就沒有對齊基準。

## 4. 架構

### 4.1 Engine 分層

對照策略總覽的四層驅動模型：

- **Layer 1 Prompt**（Python 組、每輪）：組 system prompt = 貫通軸 + 關係層 + Setting + ambient echo。無 LLM。
- **Layer 2 Judge**（Gemini Flash、每輪）：追蹤 stage / mode / 張力 / fire_release / basin_lock。
- **Layer 3 Composer**（Opus 4.7、session 邊界）：bake snapshot（21 格 + effective 關係層 + Judge principles）+ hash cache。
- **Layer 4 印象帳本**（多模型、多時機）：
  - 候選印象產出：Gemini Flash（每輪，附在角色主回應之後）
  - Ambient echo retrieval：Gemini Flash（每輪，per-turn top-3 symbol 排序）
  - **Composer pre-selection**：Gemini Flash（每次 bake 前，從 ledger 挑 top-N 相關印象餵 Opus）
  - Rubric 評分：**Opus 4.7**（session 結束，品質閘門）
  - 帳本寫回：Python（session 結束，append-only）

#### 4.1.1 Composer pre-selection（cost 優化）

策略總覽原本的 Retrieval Worker 只設計給 per-turn ambient echo。本 spec 將其**對稱延伸為 per-bake composer pre-selection**——因為 ledger 會隨時間膨脹，若每次 bake 都把整本 ledger 塞給 Opus，成本會線性上升。

Pre-selection 運作：

```
[Composer bake 前]
  active_symbols ← 當前 Active 關係層的 symbols
  setting_symbols ← 當前 Setting 的 symbols
  query_symbols = active_symbols ∪ setting_symbols
       ↓
  [Flash 讀整本 ledger]
  對每條印象打分：score = 命中 query_symbols 的數量 × 承重分數
       ↓
  取 top-N（N = 15，可調）
       ↓
  Opus 的 Composer input =
    貫通軸 + 當前關係層 + Setting + top-15 印象（不是整本 ledger）
```

這不改變策略總覽的架構原則，是**同一個 retrieval 機制換 scope**（per-turn → per-bake）。

### 4.2 模型分工

```
角色演出                → Gemini 2.5 Flash   大量、高頻
Judge                   → Gemini 2.5 Flash   每輪、結構化
Ambient echo retrieval  → Gemini 2.5 Flash   每輪、symbol 命中排序
Composer pre-selection  → Gemini 2.5 Flash   session 邊界、ledger 預篩 top-N
Rubric 評分             → Claude Opus 4.7    session 結束、品質閘門
Composer                → Claude Opus 4.7    session 邊界、抽象組裝
```

呼叫方式：
- Gemini：`google-genai` Python SDK（**不用 CLI**，因為要 batch + async）
- Opus：`anthropic` Python SDK

品質閘門（Rubric）和組裝者（Composer）兩個需要抽象推理的環節都交 Opus，其餘結構化任務交 Flash。

## 5. 檔案落盤 Schema（最重要）

這是**共同真相源頭**。Claude 用 Read 讀這些檔；dashboard 用 fetch 讀同一份檔。

### 5.1 每次實驗的目錄結構

```
runs/<exp_id>/<run_timestamp>/
├── config.yaml                 # 複製當次用的 experiment config（保證可重現）
├── conversation.md             # 對話全文（markdown，人類可讀）
├── conversation.jsonl          # 對話全文（結構化，dashboard 用）
├── turns/
│   ├── turn_001.yaml           # 每輪的完整 snapshot
│   ├── turn_002.yaml
│   └── ...
├── judge/
│   ├── state_history.jsonl     # 每輪結束的 Judge state
│   └── raw_responses.jsonl     # Gemini Judge 原始回應（debug 用）
├── composer/
│   ├── initial_snapshot.yaml   # session 開始的 bake 結果
│   ├── final_snapshot.yaml     # session 結束的 re-bake
│   ├── input_selection.yaml    # pre-selection 結果（ledger 哪些印象被選進）
│   └── cache_key.txt           # hash(貫通軸 + 關係層 + Setting + ledger_version)
├── ledger/
│   ├── candidates.jsonl        # 每輪產出的候選印象句
│   ├── rubric_scores.jsonl     # rubric 評分細節（每條六維度 + 總分）
│   └── accepted.yaml           # 通過閾值、寫入帳本的印象
└── meta.yaml                   # 總結（耗時、tokens 用量、終止原因）
```

### 5.2 `turns/turn_N.yaml` schema

**候選印象的產出方式**：Gemini 角色採結構化輸出，prompt_assembler 要求回應格式為：

```
<主回應正文>

---IMPRESSIONS---
- text: "..."
  symbols: [...]
- text: "..."
  symbols: [...]
```

Parser 切出主回應寫入 `response.content`，切出印象 block 寫入 `candidate_impressions`。角色自己決定哪些瞬間值得登錄（自覺的策略總覽 Layer 4「Main LLM 自合成情緒」原則）。


```yaml
turn: 8
speaker: counterpart  # protagonist | counterpart | scripted
timestamp: 2026-04-21T11:30:15Z
prompt_assembled:  # Layer 1 組出來的 system + user
  system: |
    <完整的 system prompt>
  user: |
    <上一輪對話、當前指示等>
  tokens:
    system: 1842
    user: 316
response:
  content: |
    你從來沒有找過我。不是他不讓你找。是你沒有找。
  raw: |
    <Gemini 完整回應，包含任何 think / reasoning trace>
  tokens_out: 35
  model: gemini-2.5-flash
  latency_ms: 412
scripted_injection: true  # 這輪是硬注入還是 model 自然產
candidate_impressions:    # Layer 4 Main LLM 自合成
  - text: "她的沉默在這一刻比任何辯解都沉"
    triggered_by: counterpart 的指控
    symbols: [沉默, 辯解, 愧疚]
judge_state_after:
  stage: 觸發
  mode: 基線 → 對峙
  tension: 0.82
  fire_release_armed: true
  basin_lock: false
```

### 5.3 `composer/input_selection.yaml` schema

紀錄 pre-selection 選了哪些印象進 Composer、為什麼選。**這個檔是 cost 審計 + debug 入口。**

```yaml
bake_timestamp: 2026-04-21T11:45:30Z
pre_selected_by: gemini-2.5-flash
ledger_version: 7
selection_criteria:
  active_symbols: [沉默, 愧疚, 靠近, 拒絕]   # 從當前 Active 關係層抽
  setting_symbols: [壓低, 懸吊, 剝除]        # 從當前 Setting 抽
  top_n: 15
  scoring_formula: symbol_hit_count * rubric_score
selected_impressions:
  - id: imp_001
    symbol_hits: 3
    matched_symbols: [沉默, 愧疚, 靠近]
    rubric_score: 16
    score: 48
  - id: imp_007
    symbol_hits: 2
    matched_symbols: [拒絕, 愧疚]
    rubric_score: 17
    score: 34
  # ... 共 15 條
full_ledger_size: 127  # 篩選前帳本總數
tokens_sent_to_composer: ~9800  # Composer input 總 tokens（篩選後）
```

### 5.4 `ledger/rubric_scores.jsonl` schema

每行一個候選印象的評分：

```json
{
  "candidate_id": "cand_008_001",
  "text": "她的沉默在這一刻比任何辯解都沉",
  "scored_by": "claude-opus-4-7",
  "dimensions": {
    "張力對拉": 3,
    "具體度": 2,
    "向性": 3,
    "未說重量": 3,
    "關係指向": 3,
    "情感克制": 2
  },
  "total": 16,
  "threshold": 14,
  "accepted": true,
  "reasoning": "此句承載了未說的重量..."
}
```

### 5.5 帳本 `A_x_B.impressions.yaml`（跨實驗累積）

不在 `runs/` 下，而是在 `ledgers/<A>_x_<B>.yaml`。跨 session 累積：

```yaml
relationship: 母親_x_兒子
created: 2026-04-21
last_updated: 2026-04-21T11:42:03Z
ledger_version: 7  # 每次寫入 +1，用於 composer hash cache
symbols_index:  # 反向索引，retrieval 用
  沉默: [imp_001, imp_003, imp_007]
  愧疚: [imp_001, imp_002]
  ...
impressions:
  - id: imp_001
    text: "她的沉默在這一刻比任何辯解都沉"
    symbols: [沉默, 辯解, 愧疚]
    from_run: mother_x_son_hospital_v3_001/2026-04-21T11:30:00Z
    from_turn: 8
    rubric_score: 16
    created: 2026-04-21T11:42:03Z
  - ...
```

### 5.6 關鍵不變量

- **一個 `run` 永遠不覆寫**：每次跑實驗建新 timestamp 子目錄。歷史不可變。
- **帳本增量更新**：只 append，不刪。ledger_version 單調遞增。
- **Cache key 由 ledger_version 參與**：帳本一更新，hash 自動 miss，composer re-bake。

## 6. Dashboard

- **單 HTML 檔 + vanilla JS**（no framework, no build, no server）
- 開啟方式：`open dashboard/index.html` 後拖一個 `runs/<exp_id>/<timestamp>/` 資料夾進去
- 或：`python -m http.server` 在 project root 跑，`http://localhost:8000/dashboard/`

### 6.2 畫面骨架（優先序）

- 讀 `runs/` 所有目錄
- 列表：exp_id / timestamp / 終止原因 / turn 數 / 通過帳本的印象數

### 6.3 Debug 優先項（依照你 v1 的痛點 c > b）

## 8. 已知風險

### 8.1 Rubric 評分已用 Opus 4.7（原風險已解除）

原本顧慮 Flash rubric 通過率虛高——這一版已把 rubric 升級到 Opus 4.7 做品質閘門，風險基本解除。代價：每 session 多 ~$0.18 成本（見 §8.4）。

反向風險：**閘門太嚴**，好印象被刷掉，帳本累積太慢。

**監控**：每跑 10 個實驗後，手動抽查 3 個被**拒絕**的候選印象，看判斷合不合理。

**應急**：
- 太嚴 → 調低六維度閾值（預設 14，可降到 12 或 10）
- 太鬆 → 調高閾值（14 → 16 → 18）

### 8.1b Composer pre-selection 的相關性風險（新增）

**症狀**：Flash 從 ledger 挑的 top-N 印象跟當前關係層/Setting 其實沒那麼相關，導致 Composer bake 出的 snapshot 偏掉。

**監控**：每次 bake 後看 `composer/input_selection.yaml`——被選中的印象 symbols 是否跟 active_symbols / setting_symbols 有意義地重疊。

**應急**：
- Pre-selection 改用 Opus（成本回升，但相關性最好）
- 改 scoring formula（例如加權 symbol 命中 vs 承重分數）
- 增加 top_n（15 → 20 → 25），容忍更多雜訊但提高 recall

### 8.2 Judge Flash 張力偵測可能糊

**症狀**：fire_release 該觸發沒觸發，或亂觸發；stage 轉換遲鈍。

**監控**：對照人工讀對話的判斷，dashboard 會顯示每輪 Judge state history。

**應急**：Judge 改 Gemini Pro 2.5 或 Opus。

### 8.3 角色 Flash 情緒動詞穿透可能弱

**症狀**：角色回應變得平板、描述性（旁白身體而非體現）。v1 qwen3 覆轍。

**監控**：盲審對話（沿用你 baseline 的四維度 rubric + 「這個角色站住了嗎」的一句話判斷）。

**應急**：角色改 Gemini Pro 2.5 或 Opus。成本會上升，但這是核心品質，不能讓步。

### 8.4 Opus 成本估算（含 pre-selection 優化）

兩個 Opus 使用點：**Composer bake + Rubric 評分**。

**Composer bake 成本演算（pre-selection 前後對比）**：

| 情境 | Input tokens | Output tokens | 單次 cost |
|------|-------------|---------------|-----------|
| Pre-selection **前**（整本 ledger 塞入） | ~25k | ~2.5k | ~$1.5 |
| Pre-selection **後**（Flash 篩 top-15） | ~10k | ~2.5k | ~$0.40 |

Session **start** 的 bake → 通常 cache hit → ~$0
Session **end** 的 re-bake → 幾乎總是 miss（帳本剛更新）→ ~$0.40

**Rubric 評分成本**：每 session 結束評分 2-5 條候選，每條 ~1k in + ~300 out → 每 session ~$0.15-0.20

**合計**：

| 項目 | per session | 100 個實驗 |
|------|-------------|-----------|
| Composer re-bake（優化後） | ~$0.40 | ~$40 |
| Rubric 評分（Opus） | ~$0.18 | ~$18 |
| **Opus 合計** | **~$0.58** | **~$58** |
| Gemini Flash（角色 + Judge + pre-select + retrieval） | ~$0.05 | ~$5 |
| **Engine 總成本** | **~$0.63** | **~$63** |

跟 pre-selection 前的 ~$1.5/session（100 實驗 ~$150）相比，**省 ~58%**。

若要再降：
1. Composer 換 Sonnet 4.6（品質會掉，21 格結構需抽象組裝能力）
2. Pre-selection top_n 從 15 降到 10（input 更少，相關性可能變差）
3. 六維度 rubric 改成三維度（input/output tokens 減半）
